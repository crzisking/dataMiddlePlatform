# 中台架构

> 本文档聚焦中台**平台层**：后端框架、认证权限、模型选择、部署与规模、对话管理。
> RAG 见 [RAG系统.md](RAG系统.md)，TextToSQL 见 [TextToSQL.md](TextToSQL.md)。

---

## 一、后端技术选型

| 层 | 选型 | 理由 |
|---|---|---|
| **Web 框架** | **FastAPI** + Uvicorn | Python 异步、SSE 流式输出原生友好、自动 OpenAPI 文档对接前端 |
| **Agent / 编排** | **LangGraph** | 路由决策（SQL/RAG/混合）是有状态图，控制力强、可观测、好调试 |
| **LLM 接入** | 通义千问 + DeepSeek，统一走 **OpenAI 兼容接口** | 两家均提供 OpenAI 格式 API，用 `litellm` 或自封装做模型路由 / 降级 |
| **任务队列** | **procrastinate**（PostgreSQL 后端） | 文档解析 / embedding 等异步任务，不阻塞请求；**复用现有 PostgreSQL，无需 Redis 等额外中间件** |
| **ORM / 迁移** | SQLAlchemy + Alembic | 中台自身元数据表（对话历史 / 文档管理 / 切割配置等） |
| **可观测** | **Langfuse**（自托管） | LLM 调用追踪、token 成本、debug |

## 二、认证与权限（已确定：一期极简）

- **不做单点登录（SSO）**：中台是内部后台，直连使用。
- **不分用户、不做 API 鉴权**：一期所有人共享访问，接口无 token / API Key 认证。
- **暂不做部门级权限隔离**，但**数据结构上预留**：
  - 文档 chunk、数据查询结果仍带「部门 / 业务标签」元数据字段
  - 检索与查询层预留按标签过滤的入口
  - 后期需要权限/隔离时无需改表结构，只需启用过滤规则与接入鉴权

## 三、模型选择（已确定）

- **由用户在对话中自行选择模型**（通义千问 / DeepSeek）。
- 前端对话框提供模型切换；接口请求带上所选模型标识，后端按标识路由到对应 API。

## 四、部署方式

- **一期：部署在 Windows 服务器**（FastAPI + procrastinate worker + PostgreSQL）
- **后期：迁移到 Docker**（容器化编排）
- 设计上保持环境无关：配置走环境变量 / 配置文件，不绑定 Windows 特有路径，便于后续平滑迁移容器。

### 规模量级（已确定）
- 用户数：**500 ~ 1000**
- 并发：**200 ~ 300**

### 规模带来的设计影响
- 请求大多是 **LLM/embedding API 的 I/O 等待**（FastAPI 异步对高并发 I/O 友好），单机可承接，但要注意：
  - **上游 API 限流**：通义 / DeepSeek 的 QPS、并发上限可能先于本机成为瓶颈，需做**请求排队 / 限流 / 重试降级**。
  - **Uvicorn 多 worker**：Windows 下用多进程 worker 提升并发承载。
  - **procrastinate worker 数**：文档解析 / embedding 入库是 CPU/IO 混合任务，按文档吞吐量配置 worker 并发，避免与 Web 进程抢资源。
  - **注意 PG 负载**：任务队列也落在 PostgreSQL，高并发下需关注 PG 连接数与队列表的读写压力。
  - 200~300 并发下，**Windows 单机偏紧**，需压测验证；这也是后期迁 Docker（便于横向扩容）的动因之一。
  - ⚠️ **Windows 事件循环约束**：psycopg(异步) 与 procrastinate 在 Windows 上必须用 `SelectorEventLoop`（已在 `app/main.py` 顶部设置），而它有 **~512 套接字上限**。200~300 并发接近该量级时需留意，**是迁移 Docker(Linux) 的又一动因**（Linux 无此限制）。

## 五、对话管理（已确定）

存在两类客户端，对话存储**分端处理**：

| 客户端 | 对话历史存储 |
|---|---|
| **TS+Vue 桌面端** | 存**本地 SQLite**（已有），单机工具够用，不入服务端 |
| **Web 网页端** | 存**服务端 PostgreSQL**（会话表 + 消息表），支持多轮上下文、多端访问、历史回看 |

### 定位：中台是纯服务，不做用户/审计功能
- **用户身份由调用方记录**：上传等入口借用其他系统的代码，用户在那边记录；中台本身**不做用户管理、不做按人审计**。
- 中台只负责提供 RAG / TextToSQL 服务，保持职责单一。
- 可保留**应用级运行日志**（请求/错误/耗时，用于排障），但不绑定用户身份、不作为合规审计用途。

### 设计要点
- 网页端接口需带 **会话 ID**，以支撑多轮上下文。
- 桌面端可走无状态接口（上下文由本地组装后随请求带上）。
- **历史是唯一真相，不与 Agent 框架的内部记忆重复**：服务端不使用 LangGraph checkpointer 持久化，Agent 无状态，每次按上述历史拼成消息列表调用（见「六、Agent 设计」方案 B）。

## 六、Agent 设计（已确定：多轮对话 + 工具调用）

核心交互是一个**多轮对话 Agent**，而非写死的 if/else 路由：

- **工具调用模式**：RAG 检索、TextToSQL 查询等能力都做成**工具（skill）**，由模型根据用户问题**自主决定调用哪个、调几次、如何综合**，可多步循环（调用 → 看结果 → 再调用 → 作答）。
- **实现**：`langchain.agents.create_agent`（建在 LangGraph 上的工具调用 Agent；旧版叫 `langgraph.prebuilt.create_react_agent`，已迁移）。
- **可扩展**：新增一个能力 = 在 `app/services/agent/tools.py` 加一个 `@tool` 并登记到 `TOOLS`，无需改主流程。
- **多轮记忆（方案 B，已定）**：**Agent 无状态**，不依赖 LangGraph checkpointer 持久化。
  - **以自建的「会话表 + 消息表」（PG）为唯一真相**，每次请求把「历史消息 + 本轮」拼成完整列表喂给 Agent。
  - 历史来源分两端，但都归一成「准备好一个消息列表」：网页端由服务端从 PG 取（P4）、桌面端由客户端本地组装后随请求带上。
  - 好处：会话列表 / 重命名 / 搜索等产品功能可控；无双写、无两份真相。聊天场景用不到 Agent 断点续跑，故不引入 checkpointer。
- **模型可选**：按用户所选模型构建对应 Agent（见「三、模型选择」）。
- **流式输出（SSE）**：对话接口分两个——`/chat`（一次性返回）与 `/chat/stream`（SSE，逐 token 推送）。一期先流最终答案；工具进度提示（"正在查询数据库…"）作为 P4/P6 增强。**契约已定，前端可据此对接。**

当前已落地的工具（骨架占位，待对应阶段接真实现）：

| 工具（skill） | 能力 | 接入阶段 |
|---|---|---|
| `search_knowledge_base` | 知识库检索（RAG） | P4 |
| `query_business_database` | 业务数据库查询（TextToSQL，SQL Server） | P5 |
