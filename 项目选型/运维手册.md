## 一、系统组成（运行时有哪些东西）

| 组件 | 是什么 | 在哪 | 挂了会怎样 |
|---|---|---|---|
| **API 服务** | FastAPI(uvicorn)，对外 HTTP 接口 | 本机:8000 | 整个服务不可用 |
| **Worker** | procrastinate 后台进程，处理上传后的解析入库 | 本机，常驻 | 上传的文档不会被解析（一直 pending），其余正常 |
| **PostgreSQL + pgvector** | 存：文档元数据、向量、对话、切割配置、语义层、**任务队列** | 远程 `192.168.20.99` / 库 `data_platform` | 入库/检索/会话/队列都受影响（但 API 仍能启动，见容错） |
| **MinIO** | 存上传的原始文件 | `192.168.120.198:9000` / 桶 `temp` | 上传/下载原件失败 |
| **SQL Server（业务库）** | TextToSQL 只读查询用。⚠️**生产库，绝不写/不建对象/不跑重查询** | `192.168.20.6` / 库 `ICAS_FPC`（pymssql，TDS 7.0） | TextToSQL 查不了；RAG 不受影响 |
| **通义 / DeepSeek** | 大模型 + 向量化 embedding + rerank（云 API） | 公网，靠 `.env` 的 key | 问答/入库向量化失败（有重试） |

> **任务队列也在 PostgreSQL 里**（procrastinate 用 PG，不需要 Redis）。
> **前端**是独立仓库 `D:\workSpace\tmbomweb`（Vue3），门户挂载式，本手册只讲后端。

## 二、端口与数据流

- API：`8000`（HTTP）。前端开发：`8991`。MinIO：`9000`。PG：`5432`。SQL Server：`1433`。
- **RAG 问答数据流**：前端 → `POST /api/v1/chat` → Agent（LangGraph）→ 工具检索 PG 向量库 → 大模型生成 → 返回答案 + 来源（MinIO 现签下载链接）。
- **入库数据流**：上传 → 存 MinIO + 登记 documents(pending) → 投 procrastinate 队列 → Worker：取原件→解析→切割→向量化→写 chunks→done。
- **TextToSQL 数据流**：问题 → 检索语义层(PG schema_docs) → 大模型生成 SQL → 安全护栏校验 → 只读查 SQL Server → 返回结果。

## 三、首次部署（新机器，从零）

> 正式环境后端部署在 `192.168.120.174`（独立机；PostgreSQL 在 `192.168.20.99`，两台分开 —— 后端要能跨网段连到 PG/MinIO/SQL Server）。

1. **装 Python 3.12 + uv**。进项目目录，`uv sync` 装依赖（会建 `.venv`）。
2. **配 `.env`**：复制 `.env.example` 为 `.env`，逐项填真实值（第四节有每项说明）。**正式环境务必 `APP_DEBUG=false`**。
3. **PostgreSQL 准备**（一次性，详见 [数据库与迁移SOP.md](数据库与迁移SOP.md)）：
   - 建库：连 PG 执行 `CREATE DATABASE data_platform;`
   - 装并启用 pgvector：`CREATE EXTENSION IF NOT EXISTS vector;`（Windows 无现成包要编译，见 SOP 附录）
4. **建表**：`scripts\ops.ps1 migrate`（= `alembic upgrade head`）。
5. **建队列表**（一次性）：`uv run procrastinate --app=app.workers.queue.app schema --apply`。
6. **自检 SQL Server**（若用 TextToSQL）：`scripts\ops.ps1 mssqlcheck`。
7. **启动 API + Worker**（见第五节）。
8. **灌知识库**：`scripts\ops.ps1 ingest <文件夹> [类型]` 批量入库，或用管理页上传。

## 四、配置详解（`.env` 每一项）

> 密钥**只在 `.env`**（不进 git）；`.env.example` 放占位符。改完**重启 API/Worker** 才生效。

**应用**
| 键 | 说明 |
|---|---|
| `APP_ENV` | 环境标识(dev/prod)，仅记录用 |
| `APP_DEBUG` | **正式环境设 `false`**：日志降到 INFO、关 SQL echo。`true` 时日志含 DEBUG、SQL 全打 |
| `APP_HOST` / `APP_PORT` | 监听地址/端口（默认 0.0.0.0:8000） |

**PostgreSQL（我们自己的库）**
| 键 | 说明 |
|---|---|
| `PG_HOST/PORT/USER/PASSWORD/DATABASE` | 连接信息。代码内部派生三种连接串（异步/Alembic/队列），不用手配 |

**SQL Server（业务库，TextToSQL）**
| 键 | 说明 |
|---|---|
| `MSSQL_HOST/PORT/USER/PASSWORD/DATABASE` | 连接信息。**留空则 TextToSQL 相关功能自动跳过**（不报错） |
| `MSSQL_TDS_VERSION` | **2008 必须 `7.0`**（新协议跟老库 TLS 握手失败）。代价：date/datetime2 读成字符串（值对） |
| `MSSQL_TIMEOUT` | 单条查询超时（秒） |

**MinIO**：`MINIO_ENDPOINT/ACCESS_KEY/SECRET_KEY/BUCKET/SECURE`。

**大模型**
| 键 | 说明 |
|---|---|
| `QWEN_*` / `DEEPSEEK_*` | base_url / api_key / 默认模型 |
| `QWEN_MODELS` / `DEEPSEEK_MODELS` | **可选模型白名单**（逗号分隔）。前端下拉来源（`/meta/models`）。当前 `qwen3.7-max,qwen3.7-plus` / `deepseek-v4-pro` |
| `EMBEDDING_MODEL` / `EMBEDDING_DIM` | 向量化模型/维度(1024)。**改维度要配套迁移重建向量列** |
| `RERANK_ENABLED` / `RERANK_MODEL` | rerank 开关（默认 false）/ 模型 |

**上传 / 文档类型**
| 键 | 说明 |
|---|---|
| `UPLOAD_MAX_MB` | 单文件大小上限（默认 100） |
| `UPLOAD_ALLOWED_EXTS` | 格式白名单（默认 `pdf,docx,xlsx,txt,md`；老二进制 doc/xls 不支持） |
| `DOC_TYPES` | **文档类型受控词表**（逗号分隔）。上传/筛选下拉来源（`/meta/doc-types`） |

**上线硬化（P8）**
| 键 | 说明 |
|---|---|
| `LLM_MAX_RETRIES` / `LLM_TIMEOUT` | LLM/embedding 调用重试次数 / 超时（429/超时自动退避重试） |
| `MAX_CONCURRENT_REQUESTS` | 同时处理请求数上限，超过返回 503（默认 400，压测后调） |
| `DB_POOL_SIZE` / `DB_MAX_OVERFLOW` | DB 连接池常驻/溢出数 |
| `DB_STATEMENT_TIMEOUT_MS` | 单条 SQL 超时（毫秒，默认 30000），防慢查询挂死连接 |

> 切割参数**不在 .env**：按 doc_type 存 `chunk_configs` 表，用管理页或 `PUT /chunk-configs` 调。

## 五、启动 / 停止

**开发（热重载）**
```powershell
powershell -ExecutionPolicy Bypass -File scripts\ops.ps1 api      # 启 API（带 --reload）
powershell -ExecutionPolicy Bypass -File scripts\ops.ps1 worker   # 启 Worker（另开一个终端）
```

**生产（不要 --reload）**
```powershell
# API：不带 --reload（reload 是开发用的，会监视文件、占资源）
uv run uvicorn app.main:app --host 0.0.0.0 --port 8000
# Worker：
uv run procrastinate --app=app.workers.queue.app worker
```
- **API 和 Worker 是两个独立进程，分别启动**。上传文档要两者都在跑（或用批量入库脚本，它直接入库、不经队列、不需 Worker）。
- **停止**：对应终端 `Ctrl+C`。
- **生产常驻**：用 **Windows 服务 / 计划任务**守护这两个进程，掉了自动拉起（后期迁 Docker 用容器编排）。
- 多并发：可考虑 uvicorn 多 worker（`--workers N`），但 Windows 的 SelectorEventLoop 有 ~512 套接字上限，真要扛 200~300 并发建议迁 Docker(Linux)。
- 启动后自检：`scripts\ops.ps1 health`，或浏览器开 `http://127.0.0.1:8000/docs`。

## 六、日常运维操作

| 操作 | 怎么做 |
|---|---|
| 上传单个文档 | 管理页上传，或 `POST /api/v1/documents`（需 Worker 在跑才会被解析） |
| 批量入库一个文件夹 | `ops.ps1 ingest <文件夹> [类型]`（直接入库，不需 Worker） |
| 查文档列表/状态 | 管理页「文档管理」，或 `GET /api/v1/documents`（支持 类型/状态/文件名/时间 过滤） |
| 重新入库（改了切割配置后） | 重新上传该类文档（`ingest()` 可重入，会先删旧 chunk） |
| 改某类文档切割策略 | 管理页「切割配置」，或 `PUT /api/v1/chunk-configs/{doc_type}`（**改完要重新入库才生效**） |
| 把业务视图接入 TextToSQL | 填 `scripts/register_views.py` 后 `uv run python scripts/register_views.py`；拉字段用 `ops.ps1 scaffold <视图名>`。**详见 [TextToSQL.md](TextToSQL.md) 第八节** |
| 加/减可选模型 | 改 `.env` 的 `QWEN_MODELS`/`DEEPSEEK_MODELS`，重启 API |
| 加/减文档类型 | 改 `.env` 的 `DOC_TYPES`，重启 API |
| 开/关 rerank | 改 `.env` 的 `RERANK_ENABLED`，重启 API |
| 改表结构 | 改 `app/models/` → `ops.ps1 makemigration "说明"` → **检查脚本** → `ops.ps1 migrate` |

> ⚠️ 老格式 .doc/.wps/.xls 不支持解析，先用 Word/WPS 另存为 .docx/.xlsx 再入库。

## 七、监控与健康检查

- **进程存活 / DB 连通**：`ops.ps1 health`（含 PG + SQL Server），或 `GET /api/v1/health`、`/health/db`、`/health/mssql`。
- **SQL Server 单独自检**（不用启 API）：`ops.ps1 mssqlcheck`。
- **入库进度**：documents 表的 `status`（pending→parsing→embedding→done/failed）和 `error` 字段；管理页也能看 + 自动轮询。
- **任务队列**（PG 里）：
  ```sql
  SELECT id, task_name, status, attempts FROM procrastinate_jobs ORDER BY id DESC LIMIT 20;
  ```
  status=todo 堆积 → Worker 没跑或处理不过来；status=failed → 看 error。
- **知识库规模**：
  ```sql
  SELECT count(*) FROM documents WHERE is_active;              -- 文档数
  SELECT count(*) FROM document_chunks;                        -- 切块数
  SELECT name, chunk_count FROM documents WHERE chunk_count<=2;-- 可疑(图片型/扫描件)
  ```

## 八、错误分类与日志（重点）

系统把错误分成**两类**，处理方式不同——核心原则：**给前端的只给"用户能看懂、且不泄露内部"的；细节（堆栈/SQL/驱动报错/IP）只进日志。**

### 8.1 两类错误

**① 业务错误（可预期）** —— 抛 `app/core/exceptions.py` 里的 `AppError` 子类：

| 异常 | code | HTTP | 给前端的 message（安全、人写的） |
|---|---|---|---|
| `BadRequestError` | `BAD_REQUEST` | 400 | 如"不支持的文件类型…" |
| `NotFoundError` | `NOT_FOUND` | 404 | 如"文档不存在 id=…" |
| `LLMError` | `LLM_ERROR` | 502 | "模型调用失败，请稍后重试" |
| `DatabaseError` | `DB_ERROR` | 500 | "数据访问异常" |
| `ExternalServiceError` | `EXTERNAL_ERROR` | 502 | 如"业务数据库连接失败"（**不含驱动/IP 细节**） |

这些**会原样返回前端**（前端按 code 处理），并记 **WARNING** 日志。

**② 意外异常（不可预期）** —— 任何没接住的 `Exception`：
- 全局兜底处理器用 **`logger.exception` 把完整堆栈写进日志**；
- 对前端**只回笼统** `{"code":"INTERNAL_ERROR","message":"服务异常，请稍后重试"}`（500），**绝不泄露内部细节**。

**其它**：
- 请求参数校验失败 → `VALIDATION_ERROR`(422)，detail 带具体哪个字段错（安全、有用）。
- 框架级 HTTP 错误（404 路由不存在等）→ `HTTP_ERROR`。
- 并发超限 → `BUSY`(503)，"服务繁忙，请稍后重试"。

### 8.2 统一返回结构
所有错误都返回：`{ "code": "...", "message": "...", "detail": ... }`。
code 全集：`VALIDATION_ERROR / BAD_REQUEST / NOT_FOUND / LLM_ERROR / DB_ERROR / EXTERNAL_ERROR / HTTP_ERROR / BUSY / INTERNAL_ERROR`。

### 8.3 哪些只进日志、不回前端（已落实）
- **完整堆栈**：意外异常 `logger.exception`（exceptions.py 兜底处理器）。
- **TextToSQL 生成的 SQL**：校验/执行失败时，原始 SQL 进日志（`query.py`），前端只看到"未能生成安全的查询/查询执行失败"。
- **SQL Server 原始驱动报错**（含服务器 IP）：只进日志（`db.py`），前端只看"业务数据库连接失败"。
- **入库失败原因**：存 documents.error（截断 500 字），给**管理员**在文档管理页看（内部 admin 可见，不对外）。

### 8.4 日志在哪、怎么看
- 落文件：API → `logs/api.log`，Worker → `logs/worker.log`（项目根 `logs/`，已 gitignore）。同时打控制台。
- 轮转：单文件最大 **10MB**、保留 **5** 个历史（`api.log.1`…），不会无限长大。
- 级别：`APP_DEBUG=true` → DEBUG（含每条 SQL）；`false` → INFO。**正式环境用 false**。
- **请求日志**：每条请求一行 `方法 路径 -> 状态 (耗时ms)`（logger `app.request`），排障/看流量用。
- 实时跟踪：`Get-Content logs\api.log -Tail 100 -Wait`。
- ⚠️ 第三方库 `watchfiles`（热重载用）日志已被压到 WARNING，避免刷屏；生产不用 --reload 也无此问题。

> **告警**：目前错误只落日志、**无主动告警**。邮件告警（D2）规划中、后面写。上线初期需**人工定期看 `logs/api.log` 的 ERROR/堆栈**。

## 九、常见故障排查

| 现象 | 原因 | 处理 |
|---|---|---|
| 文档一直 `pending` 不变 | Worker 没启动 | 启动 Worker；任务在队列里等 |
| 上传被拒(400) | 格式不在白名单 / 超 100MB | 转成 pdf/docx/xlsx/txt/md；或调 `.env` 的 UPLOAD_* |
| 入库后 0 块 | 扫描件/图片型无文字层 | 需 OCR（技术债）；暂转可复制文字的版本 |
| `/chat` 报模型无效(400) | 传了不在白名单的模型 | 用 `.env` 白名单里的（qwen3.7-max/plus、deepseek-v4-pro） |
| 问答很慢 | 迭代检索多轮 + LLM 延迟 | 正常（多查几轮换更全的答案）；急可关复杂问题 |
| `/chat/stream` 不出字 | SSE 解析问题 | 前端按 `\r\n` 解析（已修）；确认后端在跑 |
| 返回 503「服务繁忙」 | 并发超 `MAX_CONCURRENT_REQUESTS` | 正常保护；压测后调阈值 |
| TextToSQL 说"未配置业务数据视图" | 语义层为空 | 正常（没登记视图前不查生产库）；按 TextToSQL.md 第八节接入视图 |
| 启动报 `ProactorEventLoop` | Windows 事件循环不对 | 已由 `app/core/eventloop.py` 修复；确认它被最先 import |
| `extension "vector" is not available` | pgvector 没装 | 在 PG 服务器装 pgvector（见 SOP 附录） |
| alembic 报 `UnicodeDecodeError`(cp950) | alembic.ini 写了中文 | 保持 alembic.ini 全英文 |
| autogenerate 想删 procrastinate 表 | 队列表不在模型里 | 已由 env.py 的 include_name 过滤；勿移除 |
| rerank 出错 | 通义 rerank API 异常 | 已自动回退到融合结果；可临时 `RERANK_ENABLED=false` |
| SQL Server 连不上 | 网络/账号/TDS 版本 | `ops.ps1 mssqlcheck` 看日志；确认 `MSSQL_TDS_VERSION=7.0` |

## 十、备份与恢复

- **PostgreSQL**（最重要：元数据/向量/会话/语义层）：
  ```bash
  pg_dump -h 192.168.20.99 -U postgres -d data_platform -Fc -f data_platform.dump   # 备份
  pg_restore -h 192.168.20.99 -U postgres -d data_platform data_platform.dump        # 恢复
  ```
- **MinIO**（原始文件）：`mc mirror` 或对桶做快照/同步。
- **配套备份**：DB 存的是 MinIO 对象 key，缺一方文档就不完整 → 两者要同周期备份。
- **业务库 SQL Server 不用我们备份**（是别人的生产库，我们只读）。
- 建议：用计划任务每日定时 `pg_dump` + `mc mirror`，保留 N 天。

## 十一、安全与规模

- **密钥**：只在 `.env`（不进 git）；`.env.example` 用占位符。
- **鉴权 / CORS**：一期**无鉴权、CORS 全开**，**已决定内网不收敛**（仅内网用）。**对外公网时**必须加鉴权 + 收敛 CORS。
- **业务库只读**：SQL Server `ICAS_FPC` 是生产库，TextToSQL 只读、带护栏（仅 SELECT/白名单视图/必带过滤/TOP 限行）；**绝不写、不建对象**。
- **下载链接**：MinIO 预签名 URL，限时 1 小时，不存库、用时现签。
- **规模**：目标 500–1000 用户 / 200–300 并发。瓶颈多在上游 LLM 的 QPS；本机有并发限流(默认 400)兜底。Windows SelectorEventLoop ~512 套接字上限，接近时迁 Docker(Linux)。

## 十三、离线部署实录（2026-06-26，测试服务器 CSVSPM901 / 192.168.120.174）

> 这是真实把后端部署到测试服务器的全过程 + 踩的坑。**这台是内网隔离 + 老 CPU 的特殊环境**，
> 一般机器没这么多坑，但这节是可复现的离线部署范本。

**环境特征（决定了下面所有做法）**
- 服务器 **192.168.120.174**（独立机，非 PG 那台）。CPU 是 **Intel Core 2 Duo T7700（2007）**——**不支持 x86-64-v2**。
- **连不上 github / pypi / 公网 CDN**（`irm`/`uv` 直连被防火墙重置 10054；浏览器能下载）；**但连得上通义 dashscope API**，也连得上内网 PG(20.99)/SQLServer(20.6)/MinIO(120.198)。
- 数据库是**共享的同一个 PG**(`192.168.20.99/data_platform`)——已有数据，**无需重新迁移/灌库**。

**关键坑与对策（务必记住）**
1. **CPU 太老 → numpy 必须降到 2.2.6**：numpy ≥2.3 的 wheel 要求 x86-64-v2，这台跑不了（`NumPy was built with baseline optimizations (X86_V2)`）。降到 **2.2.6**（最后一个不要求 v2、又支持 py3.13 的版本）。
2. **离线装依赖**：在**开发机**(同为 Win x64 + py3.13.3)打 wheel 包，拷过去离线装。`uv sync --offline` **不认** `UV_FIND_LINKS`、装不上；用 **`uv pip install --no-index --find-links wheels -r requirements-offline.txt`**（已验证）。
3. **uvicorn 直跑会用 ProactorEventLoop → psycopg 连不上 PG**：`uvicorn app.main:app`(无 --reload) 会自行设循环成 Proactor，覆盖掉 eventloop.py。用 **`run_prod.py`**（自建 SelectorEventLoop 驱动 uvicorn）启动：`.venv\Scripts\python run_prod.py`。
4. **Worker 用 `python -m`**：项目没装成可编辑包，`.venv\Scripts\procrastinate` 找不到 `app` 包（`Could not load app`）。用 **`.venv\Scripts\python -m procrastinate --app=app.workers.queue.app worker`**（`-m` 会把 cwd 加进 import 路径）。
5. **Swagger /docs 空白**：UI 脚本走公网 CDN，被墙 → 页面白。只是调试页，不影响使用；想测接口用 `Invoke-RestMethod` 直连。
6. **控制台中文乱码/报错**：cp950 控制台打中文日志会 UnicodeEncodeError → 已在 `logging.py` 把 stdout/stderr 重设 utf-8 修掉。

**实际步骤（复现用）**
1. 服务器装 **Python 3.13.3**（官网安装包，勾 Add to PATH）+ **uv**（github 下 zip，解压到 `C:\uv`，加 PATH）。
2. 开发机打离线包：`uv export --no-dev --no-hashes -o requirements-prod.txt` → 去掉 `-e .` 行成 `requirements-offline.txt` → `uv run --with pip --with wheel -- python -m pip wheel -r requirements-offline.txt hatchling -w wheels`（jieba 会被构建成 wheel）。
3. 整个项目（**排除 `.venv`**，含 `wheels/`、`.env`）打包拷到服务器 `C:\dataMiddlePlatform`，删 `.venv`。
4. 改 `.env`：`APP_DEBUG=false`。
5. 装依赖：`cd C:\dataMiddlePlatform` → `uv pip install --no-index --find-links wheels -r requirements-offline.txt` → 再 `uv pip install --no-index --find-links wheels "numpy==2.2.6"`（降 numpy）。
6. 启动：API = `.venv\Scripts\python run_prod.py`；Worker = `.venv\Scripts\python -m procrastinate --app=app.workers.queue.app worker`（各开一个窗口）。
7. 防火墙放行入站 8000：`New-NetFirewallRule -DisplayName "DataPlatform API 8000" -Direction Inbound -Protocol TCP -LocalPort 8000 -Action Allow`。
8. 验证：`/api/v1/health`、`/health/db`、`/health/mssql` 都 ok；`POST /api/v1/search` 能返回 hits（通义 API 通）；从别的机器访问 `http://192.168.120.174:8000/api/v1/health`。
