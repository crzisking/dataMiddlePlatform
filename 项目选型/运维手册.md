# 运维手册

> 面向部署 / 运维 / 接手的人。讲清楚：系统由什么组成、怎么装、怎么起停、日常怎么操作、
> 出问题怎么查、怎么备份。命令封装在 `scripts/ops.ps1`，数据库细节见 [数据库与迁移SOP.md](数据库与迁移SOP.md)。

---

## 一、系统组成（运行时有哪些东西）

| 组件 | 是什么 | 在哪 |
|---|---|---|
| **API 服务** | FastAPI(uvicorn)，对外接口 | 本机，端口 8000 |
| **Worker** | procrastinate 后台进程，处理上传后的解析入库 | 本机，常驻 |
| **PostgreSQL + pgvector** | 存元数据、向量、会话、任务队列 | 远程 192.168.20.99 / 库 data_platform |
| **MinIO** | 存上传的原始文件 | 192.168.120.198:9000 / 桶 temp |
| **SQL Server（业务库）** | TextToSQL 只读查询用（P5）。⚠️**生产库，绝不写、不建对象、不跑重查询** | 192.168.20.6 / 库 ICAS_FPC（pymssql，TDS 7.0） |
| **通义 / DeepSeek** | 大模型 + 向量化 + rerank（云 API） | 公网，靠 .env 里的 key |

> 注意：**任务队列也在 PostgreSQL 里**（procrastinate 用 PG，不需要 Redis）。

依赖关系：API 和 Worker 都依赖 PG、MinIO、大模型 API。PG 挂了 → 入库/检索/会话都受影响（但 API 仍能启动，见容错设计）。

## 二、首次部署（新机器/新环境）

1. **装 Python 3.12 + uv**，进项目目录，`uv sync` 装依赖。
2. **配 `.env`**：复制 `.env.example` 为 `.env`，填 PG/MinIO/通义/DeepSeek 的真实地址和密钥。
3. **PostgreSQL 准备**（一次性，详见 [数据库与迁移SOP.md](数据库与迁移SOP.md)）：
   - 建库 `CREATE DATABASE data_platform;`
   - 装并启用 pgvector（Windows 无现成包需编译，见 SOP 附录）
4. **建表**：`ops.ps1 migrate`（= `alembic upgrade head`）
5. **建队列表**：`uv run procrastinate --app=app.workers.queue.app schema --apply`（一次性）
6. 启动 API 和 Worker（见下）。

## 三、启动 / 停止

用运维脚本（PowerShell）：
```powershell
powershell -ExecutionPolicy Bypass -File scripts\ops.ps1 api      # 启动 API
powershell -ExecutionPolicy Bypass -File scripts\ops.ps1 worker   # 启动 Worker(另开一个终端)
```
- **停止**：在对应终端按 `Ctrl+C`。
- **API 和 Worker 是两个独立进程，要分别启动**。上传新文档需要两者都在跑（或用批量入库脚本，它直接入库不经队列、不需要 Worker）。
- 生产常驻：建议用 Windows 服务 / 计划任务守护这两个进程（后期迁 Docker 后用容器编排）。

启动后自检：`ops.ps1 health`（看 API + DB 是否正常），或浏览器开 `http://127.0.0.1:8000/docs`。

## 四、日常运维操作

| 操作 | 怎么做 |
|---|---|
| 上传单个文档 | `/docs` 里 `POST /api/v1/documents`（需 Worker 在跑才会被解析） |
| 批量入库一个文件夹 | `ops.ps1 ingest <文件夹> [类型]`（直接入库，不需 Worker） |
| 查文档列表/状态 | `GET /api/v1/documents`、`GET /api/v1/documents/{id}` |
| 问答测试 | `POST /api/v1/chat`，body `{"message":"..."}` |
| 检索测试 | `POST /api/v1/search`，body `{"query":"...","mode":"hybrid"}` |
| 看会话历史 | `GET /api/v1/conversations`、`.../{id}/messages` |
| 改某类文档切割策略 | `PUT /api/v1/chunk-configs/{doc_type}`（改完需对该类文档重新入库才生效） |
| 开/关 rerank | 改 `.env` 的 `RERANK_ENABLED`，重启 API |
| 改表结构 | 改 `app/models/` → `ops.ps1 makemigration "说明"` → 检查脚本 → `ops.ps1 migrate` |

> ⚠️ 老格式 .doc/.wps 不支持解析，先用 Word/WPS 另存为 .docx 再入库。

## 五、监控与健康检查

- **进程存活 / DB 连通**：`ops.ps1 health`（已含 PostgreSQL + SQL Server），或 `GET /api/v1/health`、`/health/db`、`/health/mssql`。
- **SQL Server 单独自检**（不用启 API）：`ops.ps1 mssqlcheck` 或 `uv run python scripts/check_mssql.py`。
- **应用日志**：同时打到控制台**和文件**——
  - API → `logs/api.log`，Worker → `logs/worker.log`（项目根下 `logs/`，已 gitignore）
  - 轮转：单文件最大 10MB、保留 5 个历史（`api.log.1` …），不会无限长大
  - 含每条请求、错误堆栈、入库进度；排查时 `Get-Content logs\api.log -Tail 100 -Wait` 实时跟踪
- **入库进度**：查 documents 表的 `status`（pending→parsing→embedding→done/failed）和 `error` 字段。
- **任务队列**（PG 里）：
  ```sql
  SELECT id, task_name, status, attempts FROM procrastinate_jobs ORDER BY id DESC LIMIT 20;
  ```
  status=todo 堆积 → Worker 没在跑或处理不过来；status=failed → 看 error。
- **知识库规模**：
  ```sql
  SELECT count(*) FROM documents WHERE is_active;   -- 文档数
  SELECT count(*) FROM document_chunks;             -- 切块数
  SELECT name, chunk_count FROM documents WHERE chunk_count<=2;  -- 可疑(图片型/扫描件)
  ```

## 六、常见故障排查

| 现象 | 原因 | 处理 |
|---|---|---|
| 文档一直 `pending` 不变 | Worker 没启动 | 启动 Worker；任务在队列里等 |
| 上传被拒(400) | 格式不在白名单 / 超过 100MB | 转成 pdf/docx/xlsx/txt/md；或调 `.env` 的 UPLOAD_* |
| `/chat` 报 500、日志 `model_not_found` | 传了无效模型名 | 现已改为返回 400；用 qwen-plus/deepseek-chat 等 |
| 入库后 0 块 | 扫描件/图片型文档无文字层 | 需 OCR(MinerU，技术债)；暂转可复制文字的版本 |
| 启动报 `ProactorEventLoop` | Windows 事件循环不对 | 已由 `app/core/eventloop.py` 修复；确认它被最先 import |
| `extension "vector" is not available` | pgvector 没装 | 在 PG 服务器装 pgvector（见 SOP 附录） |
| alembic 报 `UnicodeDecodeError`(cp950) | alembic.ini 写了中文 | 保持 alembic.ini 全英文 |
| autogenerate 想删 procrastinate 表 | 队列表不在模型里 | 已由 env.py 的 include_name 过滤；勿移除该过滤 |
| rerank 出错 | 通义 rerank API 异常 | 已自动回退到融合结果；可临时 `RERANK_ENABLED=false` |
| 检索结果重复同一段 | 父子切割同一父块被多块命中 | 已在 hybrid 检索按内容去重 |

## 七、备份与恢复

- **PostgreSQL**（最重要，含元数据/向量/会话）：
  ```bash
  pg_dump -h 192.168.20.99 -U postgres -d data_platform -Fc -f data_platform.dump   # 备份
  pg_restore -h ... -U postgres -d data_platform data_platform.dump                  # 恢复
  ```
- **MinIO**（原始文件）：用 `mc mirror` 或对桶做快照/同步备份。
- 二者要**配套备份**：DB 里存的是 MinIO 对象 key，缺一方则文档不完整。

## 八、关键配置（.env）与可调参数

| 配置 | 说明 |
|---|---|
| `PG_*` | PostgreSQL 连接（密钥不进 git） |
| `MINIO_*` | MinIO 连接 |
| `QWEN_API_KEY` / `DEEPSEEK_API_KEY` | 大模型密钥 |
| `MSSQL_*` | 业务库 SQL Server 连接（TextToSQL）。留空则相关功能跳过；`MSSQL_TDS_VERSION=7.0`（2008 必需）、`MSSQL_TIMEOUT` 查询超时 |
| `RERANK_ENABLED` | 是否开 rerank 精排（默认 false） |
| `UPLOAD_MAX_MB` / `UPLOAD_ALLOWED_EXTS` | 上传大小上限 / 格式白名单 |
| `EMBEDDING_DIM` | 向量维度(1024)，**改了要配套迁移重建向量列** |
| 切割参数 | 不在 .env，按 doc_type 存 chunk_configs 表，用 `PUT /chunk-configs` 调 |

## 九、安全与规模注意

- **密钥**：只在 `.env`（不进 git）；`.env.example` 用占位符。
- **鉴权**：一期**无鉴权、CORS 全开**，仅限内网使用；对外前必须加鉴权 + 收敛 CORS。
- **下载链接**：MinIO 预签名 URL，限时 1 小时，不存库、用时现签。
- **规模**：目标 500–1000 用户 / 200–300 并发。Windows 的 SelectorEventLoop 有 ~512 套接字上限，接近时需迁 Docker(Linux)。
