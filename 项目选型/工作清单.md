## 各阶段明细

### P0 · 选型 ✅（已完成）
- 技术栈、架构、各项约束已定稿（见四份选型文档）。

### P1 · 后端骨架 ✅（已完成）
- 项目目录结构、FastAPI 启动、配置管理（环境变量/配置文件）
- PostgreSQL 连接 + PGVector 扩展启用 + Alembic 迁移
- MinIO 连接封装
- LLM 接入层（通义/DeepSeek，OpenAI 兼容，按模型标识路由）
- **多轮对话 Agent 骨架**（LangGraph 工具调用 Agent + 工具注册表；工具为占位）
  - Agent 无状态（方案 B）：历史由调用方以消息列表传入
  - 对话接口：`/chat`（一次性）+ `/chat/stream`（SSE 流式，已打通）
- procrastinate（PostgreSQL 任务队列）骨架空壳
- 统一异常处理、冒烟测试、Windows 事件循环修复（SelectorEventLoop）

### P2 · RAG 入库链路 ✅（已完成）
**① 数据模型 + 迁移 ✅**
- `documents` / `document_chunks` 两表建好，含 HNSW 向量索引、外键级联
- 流程沉淀为 [数据库与迁移SOP.md](数据库与迁移SOP.md)

**② 上传限制 + 校验 ✅**
- 白名单 PDF/Word/Excel/txt/md（后缀 + 魔数双校验、挡可执行/伪装文件）
- 单文件 100MB；同名同类型自动新版（详见 [RAG系统.md](RAG系统.md)）
- 配置可经 `.env` 调整；9 个单元测试

**③ 上传接口 ✅**
- `POST /api/v1/documents`：校验 → 存 MinIO → 建文档记录(pending) → 投递异步任务
- 端到端验证通过（DB + MinIO + 队列三处确认）

**④ 异步入库任务 ✅**
- worker 取任务后：从 MinIO 取原件 → 解析（PyMuPDF / python-docx / openpyxl / txt·md）
  → 递归切割 → 通义 embedding(1024 维) → 写 `document_chunks` → 更新 `documents.status`
- 失败落 `status=failed` + `error`；扫描件无文字层会 done+0 块（OCR 留作后续增强）
- 直连测试 + worker 集成测试均通过（真实通义 embedding）
- 老格式 .doc/.xls 暂不支持解析（明确报错引导转存 docx/xlsx）

**⑤ 回查 / 列表接口 ✅**
- `GET /api/v1/documents/{id}`：单篇详情/状态（供上传后轮询进度）
- `GET /api/v1/documents`：分页列表，支持按 doc_type / status / only_active 过滤
- 真实 docx（MRP整理.docx）端到端验证通过（解析/切块/向量/biz_tags 元数据均正确）

### P3 · 切割可配置 + 中文混合检索 🔨
- 切割策略路由（递归 / 标题层级 / 表格独立）+ 父子切割开关
- 按文档类型的可配置项（落库 + 管理接口）
- 中文全文检索：jieba 分词 + PG `simple` tsvector（方案 A，见 RAG系统.md）
- 向量 + BM25 混合检索
- **无待讨论项**（方案已定）。

### P4 · RAG 问答 🔨
- 检索 → rerank（通义，可配开关）→ 拼上下文 → LLM 生成（接入 `search_knowledge_base` 工具）
- 会话存储：网页端存 PG（会话表+消息表，多轮，方案 B 唯一真相）；桌面端本地 SQLite
- 服务端按 conversation_id 从 PG 取历史填充进 Agent 调用
- 流式管道已在 P1 打通；本阶段可增强：工具进度提示（"正在检索…"）推送
- 文档版本管理（旧版可追溯）
- 💬 **本阶段跑通后，讨论「RAG 效果评测」**（有了可跑的链路才好设计评测集与指标）。

### P5 · TextToSQL 🔨
- 接入 SQL Server 只读视图 / 只读账号
- Schema 语义层（表/字段注释、术语、表关系、样例）+ schema-RAG
- SQL 生成 → 语法/权限白名单校验 → 执行 → 超时/行数护栏
- 💬 **开工前先定「语义层构建/维护」**（初始怎么生成、表结构变更谁更新）——它是本阶段的输入前提。
- 💬 **本阶段跑通后，讨论「TextToSQL 评测」**（生成 SQL 正确率怎么测）。

### P6 · Agent 编排调优 🔨
- 架构已定（多轮工具调用 Agent，骨架在 P1 已建）；本阶段做**多工具协同的调优**：
  - 工具说明 / 系统提示词打磨，让模型正确选工具、正确组合
  - 混合型问题（先查数 + 再检索标准）的多步调用验证
  - 工具调用失败 / 兜底策略
- 不再需要写死的"路由判断"——模型自主决定调用工具。

### P7 · 前端管理页 / 上传页 🔨
- 在后端 API 稳定后开发：文档上传页、文档/切割配置管理页、对话页模型切换
- 依赖 P2~P6 的接口契约定型。

### P8 · 上线前硬化 🔨
- 💬 **此阶段前定「应用日志 / 监控」**（应用日志、错误告警）
- 💬 **此阶段前定「成本控制 / 限流」**（token 用量上限、并发限流、上游 API 限流降级）
- 实现上述 + 异常兜底，为 200~300 并发做准备。

### P9 · 压测 → Docker 迁移 🔨
- 按 500~1000 用户 / 200~300 并发压测，验证 Windows 单机瓶颈
- 迁移 Docker（顺带可评估中文分词升级到 zhparser/pg_jieba 方案 B）

## 技术债（代码审查发现，已记录待办）

> 都不影响当前功能跑通；多数与 P8(上线前硬化)或迁 Docker 阶段契合。

| 事项 | 严重度 | 说明 | 计划 |
|---|---|---|---|
| 上传整文件读内存 | 中 | 100MB × 高并发内存压力 | 改流式直传 MinIO；迁 Docker 前 |
| embedding 无重试/限流 | 中 | 通义偶发错误→整篇入库 failed | 加重试+退避；P3/P4 |
| 缺依赖 DB 的集成测试 | 中 | 上传/入库/列表靠手动脚本验证 | 加测试库 + 集成测试 |
| chunk 元数据无来源信息 | 低 | meta 仅类型/部门，展示来源需回查 documents | 可冗余存 document_id/name；P4 检索时定 |

> 已修复（本次审查）：重复 import、错误返回类型、重复入库产生重复切块、
> 切块共享 meta 字典、doc/xls 白名单不一致、版本并发竞态(加唯一约束)、
> 启动强依赖 PG(改为容错启动)。
