### P3 · 检索 + 切割优化 ✅（已完成）

**① 向量检索 ✅**

**② 中文 BM25 ✅**
- jieba 分词(`tokenize.py`)：入库写 `content_tokens`；`ts` 生成列 + GIN 索引(迁移 eebf3ca13bb4)
- `retrieval.keyword_search()`；search 接口加 `mode=vector|keyword`

**③ 混合检索 ✅**
- `hybrid_search()`：向量 + 关键词两路用 RRF(倒数排名融合)合并

**④ 切割策略可配置 ✅**
- `chunk_configs` 表 + 管理接口(`GET /chunk-configs`、`PUT /chunk-configs/{doc_type}`)：每类文档配 策略/chunk大小/重叠/父块大小
- 切割策略路由：`recursive`(默认) + `parent_child`(small-to-big：小块检索、命中返回所在父块)
- ingest 按文档类型读配置切割；检索时父子切割返回父块(实测验证)
- ⚠️ **标题层级/表格独立策略未做**：依赖解析保留文档结构(标题/表格)，当前解析拍平成纯文本 → 记入技术债

### P5 · TextToSQL 🔨

> 架构级决定已拍板（见 [TextToSQL.md](TextToSQL.md)）：只读 SQL Server、无跨库 JOIN、
> TextToSQL/RAG 都做成 Agent 工具、语义层 + schema-RAG、SQL 白名单护栏。
> 本阶段按下面 A–F 六类推进，依赖顺序：**A → B →（C ∥ D）→ E → F**。
> 起步策略（已定）：**单一业务域、几张视图先打通全链路，再复制扩展**；语义层**人工梳理**（库里基本无注释）。
> 详细设计（视图-语义层方案、embedding、大数据量约束、失败点分析）见 [TextToSQL.md](TextToSQL.md) 第七节。

**语义层支持两类数据源**（已定，详见 TextToSQL.md 7.3.2）：
- **① 视图**：DBA 建好的 `v_ai_` 宽视图（已 JOIN、已翻译）—— 首选，准确率高、护栏强。
- **② 文字描述的表**：建不了视图的（如按日期分片的 `xxx_yyyymmdd` 表、过于复杂），只能用文字描述表+字段+关系，模型直接对裸表写 SQL —— 兜底，JOIN/大表风险回归，护栏要更严、需更多样例。

**⏳ 待 DBA 拍板的决策**（影响实现路线，见 TextToSQL.md 7.3.1）：
- D-a `ICAS_FPC` 是生产库还是报表/副本库？→ 决定是否 NOLOCK / 避开生产。
- D-b 能否接受预聚合（为高频汇总建定时刷新的汇总表）？→ 决定大表聚合走预聚合还是硬扛明细。

**🅰 前置准备**（地基，最先做）
- A1 ✅ SQL Server 连接打通：用 `pymssql`，连通自检 `scripts/check_mssql.py` / `ops.ps1 mssqlcheck` / `GET /health/mssql`。
  - 实测连上 **192.168.20.6 / 库 ICAS_FPC / SQL Server 2008 R2 (SP1)**，共 **3515 张表+视图**。
  - ⚠️ 踩坑：pymssql 新版 FreeTDS 用 TDS 7.1~7.4 跟 2008 做 TLS 握手会失败，**只有 7.0 能连**（已设 `MSSQL_TDS_VERSION=7.0`）。代价：`date`/`datetime2` 被降级成字符串返回（值正确，对只读 TextToSQL 够用）。详见 [TextToSQL.md](TextToSQL.md)。
- A2 ⏳ 定第一刀业务域 + 列出该域视图/表清单。〔需你定：哪个域；并确认有无现成只读视图〕

**🅱 语义层**（P5 核心，准确率全靠它）
- B1 新增语义层 ORM 模型 + 迁移（`SchemaDoc`）：含 `source_type`(view/table) 区分两类数据源；存中文含义、字段含义、关系、样例，及大表元数据 `required_filter`/`default_window`/`max_window`/`isolation`。
- B2 写脚本从 `INFORMATION_SCHEMA` 自动拉表/视图字段 → 生成待填模板（库里没注释也能半自动起步）。
- B3 人工补中文含义/关系/样例 → 入库 + 向量化。两类都支持：视图填①、裸表填②（含显式 JOIN 关系）。〔需你/业务方：补含义〕
- B4 schema-RAG：复用 `hybrid_search`，按问题检出最相关的视图/表（绝不把整库喂 LLM）。

**🅲 SQL 生成**
- C1 生成服务：问题 + 检出表的语义 + few-shot 样例 → 产出 SQL。〔依赖 B4〕

**🅳 安全护栏**（制造业硬要求）
- D1 SQL 校验器：仅 SELECT、禁多语句/禁 `;` 拼接、视图白名单、强制行数上限（TOP n）。
- D2 执行器：只读账号执行、查询超时、同步驱动丢线程池（别堵事件循环）。〔依赖 A1 D1〕

**🅴 接入 Agent**
- E1 把 `app/services/agent/tools.py` 的 `query_business_database` 桩换成真实链路（C1 → D1 → D2 → 结果回填给 Agent）。失败返回可读提示 + 记日志，不暴露原始 SQL 报错。〔依赖 C1 D2〕

## 技术债（代码审查发现，已记录待办）

> 都不影响当前功能跑通；多数与 P8(上线前硬化)或迁 Docker 阶段契合。

| 事项 | 严重度 | 说明 | 计划 |
|---|---|---|---|
| 上传整文件读内存 | 中 | 100MB × 高并发内存压力 | 改流式直传 MinIO；迁 Docker 前 |
| embedding 无重试/限流 | 中 | 通义偶发错误→整篇入库 failed | 加重试+退避；P3/P4 |
| 缺依赖 DB 的集成测试 | 中 | 上传/入库/列表靠手动脚本验证 | 加测试库 + 集成测试 |
| chunk 元数据无来源信息 | 低 | meta 仅类型/部门，展示来源需回查 documents | 可冗余存 document_id/name；P4 检索时定 |
| 标题层级/表格独立切割未做 | 中 | 依赖解析保留结构(标题/表格)，当前解析拍平成纯文本 | 先升级 parsing(emit markdown 结构)，再加这两种切割策略 |
| 扫描件 OCR 未做 | 中 | 扫描 PDF 无文字层 → 入库 0 块 | 接 MinerU 等 OCR |
| 关键词检索简繁不互通 | 低 | BM25 按字面匹配，简体↔繁体不同字搜不到；繁体 jieba 分词也差(向量路已兜底) | 加 OpenCC 繁→简归一化(入库分词+查询分词)，需重新入库 |

> 已修复（本次审查）：重复 import、错误返回类型、重复入库产生重复切块、
> 切块共享 meta 字典、doc/xls 白名单不一致、版本并发竞态(加唯一约束)、
> 启动强依赖 PG(改为容错启动)。
