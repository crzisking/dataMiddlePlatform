# TextToSQL

> 本文档聚焦 TextToSQL：接库方式、Schema 治理、安全护栏、与 RAG 的串联。
> 平台层见 [中台架构.md](中台架构.md)；检索能力复用 [RAG系统.md](RAG系统.md)。

---

## 一、能力定位

自然语言查询业务数据，且**不仅查数据库，也可检索 RAG**，支持混合型问题（先查数 + 取标准再综合）。

---

## 二、接库方式（已确定）

- **生产库建只读视图 / 只读账号**：ERP / MES / OA 的业务数据均落在 **SQL Server**，直连该生产库，但只授予只读权限并通过视图暴露。
- 既隔离误操作风险，又避免单独搭副本库的工程量；复杂查询的资源占用通过下方安全护栏控制。

### 数据分布（已确定，不存在跨库查询）
- **所有业务数据都在 SQL Server** → TextToSQL 只针对 SQL Server 生成与执行 SQL。
- **所有 RAG 数据都在 PostgreSQL（PGVector）** → 知识检索只走 PG。
- 二者物理隔离，**不会出现一条 SQL 跨两个库 JOIN** 的情况，无需跨库查询引擎。
- ⏳ 仍需路由判断「这个问题该查 SQL Server 还是检索 RAG，还是两者」——**判断/实现方式尚未确定**（见 README 待讨论清单）。

---

## 三、引擎选型

- **Vanna.ai** 框架 或 自研 LangGraph 节点
  - Vanna：专为 TextToSQL，自带 schema 训练 + few-shot
  - 自研：更可控
- 数据库驱动：`pyodbc` / `pymssql`（**SQL Server，TextToSQL 只连此库**）
  - 注：PostgreSQL（`psycopg`）是 RAG 向量库 / 中台元数据用，不参与 TextToSQL 业务查询

---

## 四、Schema 治理（决定准确率，核心）

- **不要把整库 schema 全喂给 LLM**：制造业表多列杂，必须建**语义层**：
  - 表 / 字段中文注释
  - 业务术语词典
  - 表关系
  - 典型查询样例
- 将语义信息也存入**向量库**，查询时先检索"相关的几张表"再生成 SQL（**schema-RAG**，复用 RAG 检索能力）
- ⏳ **语义层如何构建与维护尚未确定**（初始怎么生成、ERP/MES 表结构变更后谁来更新）——见 README 待讨论清单。

---

## 五、安全护栏（制造业必须）
- 只读视图 / 只读账号（见接库方式）
- 生成的 SQL 先过**语法 / 权限校验**再执行（白名单）
- 查询超时
- 结果行数上限
- 预留行级 / 部门级权限过滤（见 [中台架构.md](中台架构.md) 权限预留）

---

## 六、与 RAG 的串联（多轮对话 Agent）

不再用写死的"路由判断"，而是把 **TextToSQL 和 RAG 都做成 Agent 可调用的工具（skill）**，由模型自主决定调用哪个、调几次、如何综合：

| 问题类型 | Agent 的自主行为 |
|---|---|
| 纯数据型（"上月华东区订单量"） | 调用「业务数据库查询」工具（TextToSQL） |
| 纯知识型（"A 产线换型 SOP"） | 调用「知识库检索」工具（RAG） |
| 混合型（"对照 SOP，本批次不良率是否超标"） | 自主多步：先查数据 + 再检索标准，综合作答 |

- 实现：基于 LangGraph 的工具调用 Agent（`langchain.agents.create_agent`），支持多轮对话记忆。
- TextToSQL 在 `app/services/agent/tools.py` 中注册为 `query_business_database` 工具。
- 详见 [中台架构.md](中台架构.md) 的「Agent 设计」。
