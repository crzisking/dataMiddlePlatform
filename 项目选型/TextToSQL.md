# TextToSQL

> 本文档聚焦 TextToSQL：接库方式、Schema 治理、安全护栏、与 RAG 的串联。
> 平台层见 [中台架构.md](中台架构.md)；检索能力复用 [RAG系统.md](RAG系统.md)。

---

## 一、能力定位

自然语言查询业务数据，且**不仅查数据库，也可检索 RAG**，支持混合型问题（先查数 + 取标准再综合）。

---

## 二、接库方式（已确定）

- **生产库建只读视图 / 只读账号**：ERP / MES / OA 的业务数据均落在 **SQL Server**，直连该生产库，但只授予只读权限并通过视图暴露。
- 既隔离误操作风险，又避免单独搭副本库的工程量；复杂查询的资源占用通过下方安全护栏控制。

### 数据分布（已确定，不存在跨库查询）
- **所有业务数据都在 SQL Server** → TextToSQL 只针对 SQL Server 生成与执行 SQL。
- **所有 RAG 数据都在 PostgreSQL（PGVector）** → 知识检索只走 PG。
- 二者物理隔离，**不会出现一条 SQL 跨两个库 JOIN** 的情况，无需跨库查询引擎。
- ⏳ 仍需路由判断「这个问题该查 SQL Server 还是检索 RAG，还是两者」——**判断/实现方式尚未确定**（见 README 待讨论清单）。

---

## 三、引擎选型

- 引擎（已定）：**自研 LangGraph 节点**（不引 Vanna 黑盒，与现有 Agent/RAG 架构一致、护栏可控）。
- 数据库驱动（已定）：**`pymssql`**（基于 FreeTDS，不依赖系统 ODBC 驱动，离线 Windows 省事）。
  - 注：PostgreSQL（`psycopg`）是 RAG 向量库 / 中台元数据用，不参与 TextToSQL 业务查询。

### 实测连接结论（2026-06-25，A1）
- 业务库：**192.168.20.6 / ICAS_FPC / SQL Server 2008 R2 (SP1)**，账号 `iemis`（只读），共 **3515 张表+视图**。
- ⚠️ **TDS 版本必须设 7.0**：pymssql 自带的新版 FreeTDS 用 7.1~7.4 跟 2008 做 TLS 握手会失败（`20002 Adaptive Server connection failed`），只有 7.0 能连（配置项 `MSSQL_TDS_VERSION`）。
- 代价：TDS 7.0 不认 2008 新类型，`date`/`datetime2` 会**降级成字符串**返回（值正确，不报错）；`nvarchar(max)`/`ntext`/`image` 等大字段可能截断。对"服务端生成 SQL、只读结果转文本喂模型"的 TextToSQL 来说够用。
- 升级路径（如遇类型/大字段问题）：改用 **`pyodbc` + ODBC Driver 17 for SQL Server**（`Encrypt=no`），原生支持 2008 全部类型，但需在机器上装 ODBC 驱动。
- 自检入口：`scripts/check_mssql.py` / `ops.ps1 mssqlcheck` / `GET /api/v1/health/mssql`。

## 四、Schema 治理（决定准确率，核心）

- **不要把整库 schema 全喂给 LLM**：制造业表多列杂，必须建**语义层**：
  - 表 / 字段中文注释
  - 业务术语词典
  - 表关系
  - 典型查询样例
- 将语义信息也存入**向量库**，查询时先检索"相关的几张表"再生成 SQL（**schema-RAG**，复用 RAG 检索能力）
- ⏳ **语义层如何构建与维护尚未确定**（初始怎么生成、ERP/MES 表结构变更后谁来更新）——见 README 待讨论清单。

---

## 五、安全护栏（制造业必须）
- 只读视图 / 只读账号（见接库方式）
- 生成的 SQL 先过**语法 / 权限校验**再执行（仅 SELECT、禁多语句、只能碰 `v_ai_` 白名单视图）
- 查询超时
- 结果行数上限（`TOP n`）
- **大数据量护栏**：必带过滤列、没带则自动注入默认时间窗、超范围截断/拒绝、按需 NOLOCK、资源限额 —— 详见 7.3.1
- 预留行级 / 部门级权限过滤（见 [中台架构.md](中台架构.md) 权限预留）

## 六、与 RAG 的串联（多轮对话 Agent）

不再用写死的"路由判断"，而是把 **TextToSQL 和 RAG 都做成 Agent 可调用的工具（skill）**，由模型自主决定调用哪个、调几次、如何综合：

| 问题类型 | Agent 的自主行为 |
|---|---|
| 纯数据型（"上月华东区订单量"） | 调用「业务数据库查询」工具（TextToSQL） |
| 纯知识型（"A 产线换型 SOP"） | 调用「知识库检索」工具（RAG） |
| 混合型（"对照 SOP，本批次不良率是否超标"） | 自主多步：先查数据 + 再检索标准，综合作答 |

- 实现：基于 LangGraph 的工具调用 Agent（`langchain.agents.create_agent`），支持多轮对话记忆。
- TextToSQL 在 `app/services/agent/tools.py` 中注册为 `query_business_database` 工具。
- 详见 [中台架构.md](中台架构.md) 的「Agent 设计」。

## 七、详细设计方案（视图-语义层方案）

> 本节是第四节「Schema 治理」的落地细化，是 P5 实际怎么做的完整思路。
> 一句话方针：**不喂整库给模型，而是建一层"业务视图"，把它们的"说明书"向量化，
> 用户提问时先检索出相关视图、再让模型照着视图写 SQL。** 准确率靠"视图建得好 + 检索找得准 + 样例给得对"三件事撑起来。

### 7.1 整体设计：这个功能到底怎么运转

核心决策：**语义层里只放"整理好的业务视图"，不放 3515 张裸表**。一个业务一个业务地加视图，
不追求覆盖多少，只追求"加进来的每个都准"。这么定的理由：

- 裸表 3515 张、命名是 `iesc21h` 这种代号，检索找对表难、JOIN 关系复杂、注释工作量爆炸；
- 业务视图已经把多表 JOIN 好、代号翻译好、字段起成业务名 —— 模型只需在一张宽表上
  `WHERE`/`GROUP BY`/`SUM`，**把复杂度在建视图时一次性消化掉**，模型那端简单到不易错。
- 本质就是数仓的"语义层/指标层"做法。

**端到端数据流**（用户问一句话 → 拿到答案）：

```
用户问题（"上月各产线不良数"）
  │
  ├─① 向量化问题，从语义层(SchemaDoc 表)检索最相关的 1~3 个视图   ← schema-RAG，复用 hybrid_search
  │
  ├─② 把这几个视图渲染成"带中文注释的 DDL 名片 + 典型问法样例"，连同
  │     "当前日期"一起作为上下文，让 LLM 生成 SQL                  ← 生成
  │
  ├─③ 校验生成的 SQL：仅 SELECT、只能碰 v_ai_ 白名单视图、禁多语句、
  │     强制加 TOP 行数上限                                         ← 安全护栏
  │
  ├─④ 用只读账号执行（pymssql，同步→线程池，带查询超时）           ← 执行
  │
  └─⑤ 结果整理成清晰文本，交回 Agent，由它综合作答                 ← 回填
```

只新增**一个 ORM 模型**（语义层表 `SchemaDoc`，存在我们自己的 PostgreSQL）。
**业务库的表/视图绝不映射成 Python class** —— 我们查业务库走的是"生成 SQL 字符串 → 执行"，
不是 ORM，所以没有理由也没必要为它写 class。

### 7.2 语义层数据模型（SchemaDoc）

**一个视图 = 语义层里一行**，字段含义打包塞在这一行的 JSON 里（"按视图"粒度，不是"按字段一行"）：

| 字段 | 内容 | 作用 |
|---|---|---|
| `source_type` | `view`（建好的视图）或 `table`（只能文字描述的裸表） | 区分两类数据源，详见 7.3.2 |
| `object_name` | 视图名 `v_ai_制程不良明细`，或裸表名/表组 | 白名单校验也用它 |
| `desc` | 中文用途："制程批次的不良记录" | 检索 + 喂模型 |
| `grain` | 粒度："一行 = 一个批次" | 防模型聚合时重复计数 |
| `columns` | JSON：`[{name, type, 中文含义, 单位, 取值范围}, ...]` | 渲染 DDL 名片 |
| `relations` | 表关系（谁的哪个键 = 谁的哪个键）。**视图通常为空（已 JOIN 好）；裸表必填** | 让模型对裸表写对 JOIN |
| `samples` | 典型问法 ↔ SQL 样例（few-shot） | **最高杠杆**，让模型从"看懂"到"写对" |
| `aliases` | 业务术语别名："不良=次品=NG" | 提升检索召回 |
| `required_filter` | 必填过滤列，如 `日期` | 大表护栏：没带它就自动注入默认窗 |
| `default_window` / `max_window` | 默认时间窗 / 最大可查范围，如 近30天 / 1年 | 防裸跑全表 / 防超范围 |
| `isolation` | 读隔离级别（是否 `NOLOCK`） | 查生产库时不阻塞业务 |
| `embedding` | `Vector(1024)` | 按问题检索 |

> 加一个业务 = 往这张表加一行（或几行）。视图结构变了，改的是**数据行**，不重新部署代码。
> `required_filter` / `default_window` / `isolation` 是**大数据量场景**的关键元数据，校验器据此强制护栏，详见 7.3.1。

### 7.3 视图怎么建（给 DBA 的规范）

总原则：**一个业务主题 = 一张拉平的宽视图，让模型永远不用自己 JOIN**。

1. **粒度单一明确**：一行代表一个什么（批次?订单行?一天一条线?），写清楚。粒度混了会算错。
2. **代号翻译成名字**：人按"华东产线"问，不按 `line_id=A03` 问。视图里 JOIN 维度表把名称带出来；
   状态码用 `CASE` 翻译成"合格/不良"。
3. **列名起成业务可懂名**（中文或清晰英文）：`defect_qty AS 不良数量`。**这步做了，语义层注释量砍一半**。
4. **维度 + 度量带齐**：维度（时间/产线/产品/客户…用来筛选分组）、度量（数量/金额…用来求和计数）。
5. **时间列干净单一**：留一个明确日期列给"上月/本季"筛选，别拆成年月日三列。
6. **保持明细粒度，不预聚合**：让模型自由按天/月/线聚合；只有极高频固定问法才单建汇总视图。
7. **一主题一视图、宽度适度**：不堆 100 列巨无霸，不 `SELECT *`。
8. **命名约定 = 安全白名单**：统一前缀 **`v_ai_`**。护栏直接规定"只有 `v_ai_` 开头的可查"，
   模型碰不到其他 3515 张表。

**DBA 每建好一个视图，交付这 5 样**（即语义层那一行的内容）：
视图名 / 一句中文用途 / 粒度 / 每列(列名+中文含义+单位) / 1~2 个典型问法。
（大表视图还要多交付：必填过滤列 / 默认时间窗 / 是否 NOLOCK —— 见 7.3.1。）

### 7.3.1 大数据量约束（性能与安全，必读）

> **视图只是"存起来的一句查询"，本身不存数据**，所以底层千万级表的代价是**每次查询现付**。
> 一个业务常涉及 7~8 张、每张千万级的表，**绝不能让模型裸跑全表**——一句没好好过滤的
> `GROUP BY` 就能让多张大表全扫、拖垮库（还可能是生产库）。下面六条是大表场景的硬规矩。

**① 一张视图只 JOIN 一张大表（事实表）+ 若干小表（维度表）**
- 千万级的是事实表（交易/批次/明细），小的是维度表（产线/产品/客户对照，几百几千行）。
- **绝不在一个视图里 JOIN 两张大表**（千万 × 千万必炸）。一个业务跨多张大表 → 拆成多个视图（各挂一张大表），或走预聚合（④）。

**② 强制"必带过滤"，尤其时间窗（护栏自动注入）**
- 每个视图规定一个**必填过滤列**（通常是日期，存进 `required_filter`）。
- 模型生成的 SQL 没带时间范围 → 校验器**自动补一个默认窗口**（`default_window`，如近 30 天），不让它裸跑全表；超过 `max_window` → 截断或拒绝。
- 给模型的上下文里写明"本视图必须按日期过滤，默认近 30 天，最长 1 年"。

**③ 视图要写成"能用上索引"的样子（谓词下推）**
- 过滤列/JOIN 键**必须是大表上建了索引的列**。〔DBA 保证〕
- 视图里**别把过滤列包在函数里**（`CONVERT(date, 下单时间)` 会让索引失效、被迫全扫）——保持裸列，让 `WHERE 日期 >= ...` 下推到大表先剪枝、再 JOIN 小维度表。

**④ 聚合密集的大表 → 上"预聚合汇总"**
- `TOP n` 行数上限**救不了聚合查询**（`SUM`/`GROUP BY` 要先扫完匹配行才 TOP）。
- 高频是"按月/按线汇总"这类问题时：建一张**预聚合表**（定时 job 把明细按 天+维度 先汇总好），让 TextToSQL 查这张小汇总表，不碰千万明细；或用 SQL Server **索引视图**（物化，但对基表写入有开销、限制多，生产库慎用）。
- 本质是**用空间/维护换查询速度**，数仓标准答案。〔需决策，见下〕

**⑤ 不阻塞生产（若连的是生产 OLTP 库）**
- 视图加 `WITH (NOLOCK)` / 连接设 `READ UNCOMMITTED`（读可能稍脏，但不锁生产），存进 `isolation`；
- 或直接查**只读副本/报表库**，别压生产。〔需决策，见下〕

**⑥ 资源兜底**
- 查询超时 + 行数上限（已有护栏）之外，大表场景再加：SQL Server **Resource Governor** 给只读账号限 CPU/内存，模型写了烂 SQL 也炸不到整库。

> **护栏（D1）据此升级**：没带必填过滤 → 自动注入默认窗；超出 `max_window` → 截断/拒绝；
> 按视图的 `isolation` 决定是否 NOLOCK。

#### ⏳ 两个待 DBA 拍板的决策（影响走哪条路）
1. **`ICAS_FPC` 是生产库还是报表/副本库？** —— 决定 ⑤：要不要 NOLOCK / 避开生产。
2. **能否接受预聚合？**（为高频汇总问题建定时刷新的汇总表）—— 能：大表聚合稳；不能：只能靠"强制时间窗 + 索引"硬扛明细。

### 7.3.2 两类数据源：视图 vs 文字描述的裸表

现实里不是每个业务都能建出干净视图——有的表**建不了视图**（典型：按日期分片的 `iesc21h_20171103`
这种几百张物理表、跨度太大或结构太乱）。所以语义层**同时支持两类数据源**，用 `source_type` 区分：

| | **① 视图（`source_type=view`，首选）** | **② 文字描述的裸表（`source_type=table`，兜底）** |
|---|---|---|
| 适用 | DBA 能整理出业务视图的 | 建不了视图：分片表、结构太复杂、临时探索 |
| 语义层存什么 | 一个 `v_ai_` 宽视图的名片（已 JOIN、已翻译） | 一张/几张裸表 + 字段含义 + **显式 JOIN 关系**（`relations` 必填） |
| 谁来 JOIN | 视图里建好了，模型**不用 JOIN** | **模型自己 JOIN**（靠 `relations` + 样例） |
| 喂模型的 DDL 名片 | 单张视图的列清单 | 多张表的列清单 + 关系说明 + 完整 JOIN 的 SQL 样例 |
| 准确率 | **高**（JOIN/检索难题已消） | **较低**（JOIN 风险、选错表风险回归） |
| 大表护栏 | 同 7.3.1 | 同 7.3.1，且**更严**（裸表更大更危险，必带过滤、样例里就写死时间窗和 NOLOCK） |
| 维护 | 改视图 | 改文字描述（分片表还要告诉模型"表名带日期后缀"的规律） |

**两类共用同一套机制**：都靠 `desc + 字段中文名 + 样例 + 别名` 做 embedding 检索（7.4），
检索命中后再各自渲染对应的 DDL 名片喂给模型。区别只在**存什么、模型要不要自己 JOIN**。

**裸表路径要特别注意的点**：
- `relations` 必须写成**结构化的连接键**（`A.lot_no = B.lot_no`），不能只写一句白话，否则模型 JOIN 乱配。
- 分片表（`xxx_20171103`）：在 `desc`/样例里讲清**命名规律和怎么按日期定位到具体表**，或由 DBA 先建一个
  `UNION ALL` 的逻辑视图把分片合起来（能建的话就回到①，更省心）。
- few-shot 样例在裸表路径里**权重更高**——直接给一条"这类问题的完整多表 JOIN SQL"，是让它写对的最有效手段。

> 原则：**能建视图就走①**（一次性消化复杂度）；**实在建不了才退②**（用文字描述 + 强样例 + 严护栏兜）。
> 两条都设计进去，但①是主力，②是补充。

### 7.4 视图怎么变成向量（embedding 策略）

**注意：变成向量的不是视图里的"数据"，而是视图的"说明书文本"。** 一个视图 → 一段描述文本 → embedding → 存进 `SchemaDoc.embedding`。

关键设计 —— **"用来检索的文本" ≠ "喂给 LLM 的文本"**，因为两者目标不同：

- **检索用文本（拿去 embedding）**：偏"用户会怎么问"的语言 = 视图用途 + 字段中文名 + 典型问法 + 术语别名。
  目的是让"用户问题的向量"能匹配上对的视图。所以要尽量覆盖用户的说法（同义词、口语）。
- **喂 LLM 用文本（检索命中后才组装）**：偏"精确 DDL" = 确切视图名 + 列名 + 类型 + 注释 + 样例。
  目的是让模型写出**列名拼写正确、能跑**的 SQL。

技术上复用 RAG 整套：通义 `text-embedding-v3`（1024 维）、存 pgvector、检索直接调现成的 `hybrid_search`
（向量 + 关键词 + RRF）。检索粒度 = 一个视图一条向量。

### 7.5 哪里会不准 + 怎么解决（重点，需要想清楚）

TextToSQL 的难点**不在"存字段含义"**（那是最简单的 30%），而在下面这些。按链路逐段拆：

**A. 检索阶段：找错 / 找不全视图（最大风险，行话 schema linking）**
- 不准在哪：用户说"次品"、视图写"不良" → 向量召回不到；多个视图沾边选错；需要两个视图只召回一个。
- 怎么解：① embedding 文本里塞进"典型问法 + 术语别名"，扩大语义覆盖；② top-k 放大 + rerank 精排；
  ③ **视图方案本身就大幅降低难度**（候选从 3515 张表→十几个视图，且单视图设计让"需多视图"的概率很低）；
  ④ 维护业务术语别名表。

**B. 生成阶段：视图选对了，SQL 写错**
- 不准在哪：
  - 聚合/分组逻辑错 → **few-shot 样例是最高杠杆**，给准样例最管用；
  - 相对时间理解错（"上月""本季"）→ **把"今天的日期"注入上下文**，让模型自己换算起止；
  - 口径/公式不知道（不良"率" = 不良数/投入数）→ **要么视图里直接建好"不良率"列，要么在样例/注释里写明算法**；
  - 枚举值不知道有哪些（status 有哪些取值）→ **关键枚举值写进字段注释**。
- 怎么解：样例 + 口径前置 + 注入日期 + 枚举写清楚（都落在语义层内容里）。

**C. 执行阶段：SQL 生成了但跑挂/跑慢（大表是重灾区）**
- 不准在哪：列名拼错跑报错；**多张千万表全扫、聚合无过滤 → 拖垮库**；TDS 7.0 大字段截断（已知）。
- 怎么解：① **大数据量护栏（见 7.3.1）**——必带过滤、自动注入默认时间窗、一视图只挂一张大表、聚合走预聚合表、查询超时 + 行数上限 + 资源限额；
  ② **可选增强：把执行报错回流给模型让它自修一次**（retry-on-error）；③ 大字段问题概率低（聚合查询少碰），真碰到再考虑升级 pyodbc。

**D. 解释阶段：数据对，模型解读错**
- 不准在哪：把 NULL 当 0、把多行误总结、把字符串日期误判。
- 怎么解：结果以**清晰表格文本**回传 + 系统提示"只依据返回的数据作答，不要编造"。

**E. 语义层质量本身：根因（GIGO）**
- 不准在哪：视图粒度建错 → 聚合翻倍；注释含义写错 → 整条全错。
- 怎么解：**小步**——每加一个视图，先用一组评测问题验证它（见 F1），错了**改视图/改注释，而不是怪模型**。

**F. 维护漂移：表结构变了，视图/语义层没跟上**
- 怎么解：B2 脚本定期把视图列和语义层 diff，变更走人工复核更新。

> 一句话：**视图建得好能消掉 B 的 JOIN 难题、A 的检索难题一大半；剩下的靠样例、口径前置、护栏、评测兜。**
> 准确率是"设计 + 数据质量"问题，不是"模型够不够强"问题。
> **注意**：走"文字描述裸表"路径（7.3.2 ②）时，A 的选错表、B 的 JOIN 写错风险会**回归**，
> 所以裸表路径要靠更强的 `relations` + 完整 JOIN 样例 + 更严的过滤护栏来补，且优先考虑能不能改建成视图。

### 7.6 起步节奏

先打通**一个**业务视图的全链路（B1 建表 → B2 拉字段 → 灌一行语义 → 检索→生成→校验→执行→回填），
用评测问题验准，再照葫芦画瓢加第二个、第三个。**准确率从第一天起就高，不被未整理的烂表拖累。**

## 八、操作手册：把一个视图接入语义层（DBA 照着做）

> 适用：视图（或要文字描述的裸表）已经备好。目标：让 TextToSQL 能用它回答问题。
> 全程只动**我们自己的库和代码**，登记过程**不写业务库**（只在你最后端到端测试那一下才会只读查它）。

### 步骤 0：前提确认
- 视图已建好，且**当前的只读账号 `iemis` 能查到它**（生产库不能建对象 → 视图应建在可建对象的库/副本库；或本条走"裸表"方式）。
- 视图最好已经：多表 JOIN 好、代号翻译成名称、列名是业务可懂名（见 7.3 规范）。

### 步骤 1：打开模板
编辑 `scripts/register_views.py`，把里面的 `EXAMPLE` 换成你的真实视图（可放多条到 `VIEWS` 列表）。

### 步骤 2：逐字段填（对照下表）
| 字段 | 怎么填 | 必填 |
|---|---|---|
| `source_type` | 视图填 `"view"`；建不了视图的裸表填 `"table"` | ✓ |
| `object_name` | 视图/表名，如 `"v_ai_制程不良明细"`（也是安全白名单的依据） | ✓ |
| `desc` | 一句中文用途 | ✓ |
| `grain` | 粒度，如"一行 = 一个批次" | 建议 |
| `columns` | 每列 `{"name":精确列名,"type":类型,"desc":中文含义,"unit":单位,"values":取值}` | ✓ |
| `relations` | **视图留 `None`**（已 JOIN 好）；**裸表必填**，如 `"A.lot_no = B.lot_no"` | 裸表必填 |
| `samples` | 1~2 个"典型问法 ↔ SQL"样例（提准最有效） | 强烈建议 |
| `aliases` | 业务术语别名，如 `"不良=次品=NG"`（提升检索召回） | 建议 |
| `required_filter` | **大表必填**：必带的过滤列（通常日期列），如 `"关闭日期"` | 大表必填 |
| `default_window` / `max_window` | 默认时间窗 / 最大范围，如 `"近30天"` / `"1年"` | 大表建议 |
| `isolation` | 查生产库填 `"nolock"`（不锁业务） | 生产库建议 |
| `enabled` | 填好后改成 `True`（否则不参与检索） | ✓ |

> 把所有 `PLACEHOLDER` 都删干净——脚本里有防呆，只要还残留 `PLACEHOLDER` 就拒绝运行。

### 步骤 3：登记 + 向量化（一条命令）
```powershell
uv run python scripts/register_views.py
```
成功会打印 `已登记：v_ai_xxx（enabled=True）`。它做了两件事：写进 `schema_docs` 表 + 把说明向量化（这样检索才找得到）。**重复运行安全**：按 `object_name` 覆盖更新。

### 步骤 4：验证能问答（首次会真查生产库，建议低峰）
- 走问答接口问一个该视图能答的问题：
  ```
  POST /api/v1/chat   body: {"message": "上月各产线不良数", "model": "qwen-plus"}
  ```
- 预期：Agent 调 `query_business_database` → 检索到你的视图 → 生成 SQL → 过护栏 → 只读查询 → 返回数据。
- 查不到/报错时，日志 `logs/api.log` 里有原始 SQL 和原因，对照排查（见下）。

### 日常维护
- **改了视图内容/注释** → 改 `register_views.py` 对应条目，重跑步骤 3（覆盖更新）。
- **视图结构变了**（加/删列） → 同步更新 `columns` 再重跑，否则模型按旧字段写会出错。
- **加新视图** → 往 `VIEWS` 列表再加一条，重跑。
- **临时下线某视图** → 把该条 `enabled` 改 `False` 重跑（不参与检索，但保留记录）。

### 排查速查
| 现象 | 可能原因 | 处理 |
|---|---|---|
| 问答说"尚未配置业务数据视图" | 没登记成功 / `enabled=False` / 没向量化 | 重跑步骤 3，确认打印 enabled=True |
| 答非所问、检索到错的视图 | `desc`/`aliases`/样例 写得不贴用户问法 | 补别名、补典型问法，重跑 |
| 提示"查询必须带过滤列" | 模型没按 `required_filter` 过滤 | 在 `samples` 里给带过滤的标准样例引导 |
| 提示"查询了未授权的表" | 模型用了没登记的表 | 视图应自包含；裸表路径要把涉及的表都登记/写进 relations |
| 执行失败 | 列名/语法不符、连不上 | 看 `logs/api.log` 的原始 SQL；先 `ops.ps1 mssqlcheck` 确认连通 |
